文件结构
- .github/workflows-存储 GitHub Actions 的 CI 和 Pages 部署配置
- .git-存储本地 Git 仓库元数据，不作为业务代码目录
- build-存储 Vite 构建后的单页 html，属于生成目录
- node_modules-存储 npm 安装的第三方依赖，属于生成目录
- scripts-存储构建、资源处理等辅助脚本
- testview-存储本地预览和语法高亮验证用示例文件
- src-存储前端源码
- src/api-存储本地文件访问、阅读器配置等接口封装
- src/css-存储全局样式、布局样式和预览内容样式
- src/module-存储 Vue3 组件、业务组合函数和阅读器功能模块
- src/module/lineText-存储行文本编辑、滚动、测量和换行处理逻辑
- src/module/reader-存储阅读器状态、文件预览、worker、导航和会话逻辑
- src/module/reader/syntax-存储各语言语法高亮解析器
- src/module/reader/syntax/utils-存储语法高亮公共状态机、扫描和 token 工具
- src/module/spreadsheet-存储表格预览内部逻辑
- src/theme-存储主题状态管理
- src/utils-存储编码、文件类型、预览创建、资源重写等公共工具
- src/utils/resource-存储 html、css、srcset 等资源解析和重写工具
- 根目录配置文件-存储 package.json、vite.config.ts、tsconfig*.json、index.html、README.md、LICENSE.md、.gitignore 等项目配置和说明

拆分规范

- 普通文件不超过200行
- 原则上单一文件不能超过400行
- 代码长度计算注释行
- 如果需要拆分文件，应当使用清晰的文件夹层级，如果是公共接口就放在表层，如果是内部使用，应当放入更深的层级
  如果文件夹内部文件太多，应当根据功能创建不同的文件夹，然后将文件迁移到文件夹内，保持架构清晰

代码规范

- 如果需要连续if或其他连续逻辑，应当考虑能否优化为switch或map函数表等结构
- 应当尽量拆分功能，独立函数，普通函数保持10-50行，保证函数名和结构可以直观表现
- 应当避免过深的层级嵌套，如果超过7层，应当独立函数

注释规范

- 如果类型支持注释，必需使用vsc或vs的类型注释说明每一个函数，必需要有函数功能和参数功能1和返回类型的介绍
- 优先使用 /**\n/*\n*/这种类型的多行注释
- 如果函数或代码中有`黑魔法`或不常见或特殊的写法，应当使用单行注释在目标代码的开头
- 注释必需使用中文

git规范

- 修改前后检查状态，避免修改未提交的文件，修改后提交git保存自己的修改
- 项目可能有多人并行开发，不要提交自己未修改的文件
- .gitignore 尽量排除和项目无关的缓存，构建等
- 提交名称使用标准格式的如`fix: xxxx`,概要使用中文
